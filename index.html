Файловая система

  1. Иерархичная с единым корневым каталогом
  2. Все диски подключаются как каталоги
  3. Основные отличия от ФС в Windows
    • Не используются буквенные обозначения дисков
    • Вместо символа "\" в пути к файлы используется "/"
    • Расширение файла практически не имеет значения
  4. ФС в Linux много, но для нас это не важно

Подробно о процессах, например, тут

Демоны

Неинтерактивные процессы, которые запускаются путем загрузки в память соответствующих им программ (исполняемых файлов), и выполняются в фоновом режиме.

Не связаны ни с одним пользовательским сеансом и не могут управляться пользователем. Большую часть времени ожидают пока тот или иной процесс запросит определенную услугу, например, доступ к файловому архиву или печать документа.

Системные процессы

Являются частью ядра и всегда расположены в оперативной памяти. Не имеют соответствующих им программ в виде исполняемых файлов и запускаются при инициализации ядра системы.

Выполняемые инструкции и данные этих процессов находятся в ядре системы, таким образом, они могут вызывать функции и обращаться к данным, недоступным для остальных процессов.

Прикладные процессы

Все остальные процессы, выполняющиеся в системе. Как правило, это процессы, порожденные в рамках пользовательского сеанса работы.

Прикладные процессы Linux могут выполняться как в интерактивном, так и в фоновом режиме, но в любом случае время их жизни (и выполнения) ограничено сеансом работы пользователя. При выходе из системы все прикладные процессы будут уничтожены.

Ежедневная работа с Git (Хабрахабр)

Git: Начало работы

  1. Инициализация локального репозитория: git init
  2. Добавление к отслеживанию: git add .
  3. Закрепление изменений: git commit -am "message"
  4. Проверка состояния: git status
  5. Добавление удаленного репозитория: git remote add origin [url]
  6. Выталкивание изменений: git push

Интерактивный тренажер #1, Интерактивный тренажер #2

Git: Просмотр истории

  1. Просмотр истории: git log
  2. В одну строку: git log --pretty=oneline
  3. Кастомный вывод: git log --pretty=format:"%h %ad | %s%d [%an]" --graph --date=short
    * fa3c141 2011-03-09 | Added HTML header (HEAD, master) [Your Name]
    * 8c32287 2011-03-09 | Added standard HTML page tags [Your Name]

Интерактивный тренажер #1, Интерактивный тренажер #2

Git: Перемещения во времени

  1. Смотрим историю: git log
  2. Переходим на состояние нужного коммита: git checkout "hash"
  3. Возвращаемся к последнему состоянию: git checkout master(или HEAD)

Удобно для отката к нужному состоянию, поиска кода

Интерактивный тренажер #1, Интерактивный тренажер #2

Git: Ветки - параллельные версии проекта

  1. Создание ветки: git checkout branchName
  2. Переходим в новую ветку: git checkout branchName
  3. Создание ветки и сразу переход в нее: git checkout -b branchName

Удобно для разделения работ. Мухи и котлеты отдельно

Интерактивный тренажер #1, Интерактивный тренажер #2

Git: Ветки - слияние (merge)

  1. Переходим в ветку, в которую надо слить изменения: git checkout master
  2. Переносим изменения: git merge branchName

Git: Конфликты слияния

Конфликт будет, когда в разных ветках изменены одинаковые файлы

$ git checkout style
Switched to branch 'style'
$ git merge master
Auto-merging lib/hello.html
CONFLICT (content): Merge conflict in lib/hello.html
Automatic merge failed; fix conflicts and then commit the result.

Git: Конфликты слияния

В конфликтующих файлах появляется указание на конфликтующие строки кода и ветки

<<<<<<< HEAD
code in checked branch
=======
code in branch, which we want to merge in
>>>>>>> master

Git: Конфликты слияния

Для разрешения конфликта необходимо вручную внести изменения, приведя файл к нужному виду. После этого нужно закрепить состояние с помощью git add fileName и git commit -m "fix merge conflict"

Тонкости благополучного git-merge (Хабрахабр)

Git: Удаленные репозитории

GitHub, BitBucket, центральный сервер компании - работа в команде обычно происходит через удаленный репозиторий

  1. Клонирование репозитория: git clone remote folder
  2. Добавление удаленного реп-я: git add remote origin url
  3. Извлечение изменений: git fetch
  4. Слияние изменений: git merge origin/master
  5. Одной командой: git pull
